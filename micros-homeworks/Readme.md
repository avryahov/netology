#### Основные аргументы в пользу **Traefik**:

- **Динамическое обновление маршрутов**  
  Автоматическое обнаружение изменений (через Docker, Kubernetes, Consul и др.) без ручной перезагрузки.

- **Нативная поддержка HTTPS (Let's Encrypt)**  
  Автоматическая выдача и продление TLS-сертификатов.

- **Интеграция с Kubernetes, Docker, Consul**  
  Подходит для современных DevOps-практик без сложных конфигураций.

- **Лёгкая авторизация**  
  Через middleware: Basic Auth, JWT, OAuth2 — без Lua и внешних сервисов.

- **Наглядность и мониторинг**  
  Веб-интерфейс Dashboard, метрики Prometheus, логирование.

- **Гибкая маршрутизация**  
  По host, path, headers, тегам. Удобно при множестве сервисов.

### Почему **не** выбраны другие решения

- **NGINX** — стабильный, но устаревающий для микросервисных систем. Нет auto-discovery.
- **Kong** — требует отдельной БД, больше подходит под production API в enterprise.
- **Istio** — избыточен, требует глубокого понимания Kubernetes и Envoy.
- **Amazon API Gateway**:
    - Отличен в AWS-инфраструктуре, но:
        - Привязан к IAM и Lambda
        - Платный (по числу вызовов + трафику)
        - Не подходит для локальной разработки
- **Apigee (Google)**:
    - Подходит для крупного бизнеса, но:
        - Дорогой и тяжёлый
        - Необходима регистрация и настройка
        - Заточен под управление API, не как ingress/gateway в DevOps

### Почему **Apache Kafka (без ZooKeeper, с KRaft)**:

#### 1. **Высокая производительность и масштабируемость**
- Kafka — признанный лидер по throughput.
- Обрабатывает миллионы сообщений в секунду, масштабируется горизонтально за счёт партиций и брокеров.
- С ростом нагрузки достаточно просто добавить брокеры.

#### 2. **Удаление зависимости от ZooKeeper**
- Современные версии Kafka (2.8+ и особенно 3.x+) поддерживают **KRaft mode**, исключая необходимость в ZooKeeper.
- Это упрощает инфраструктуру, ускоряет bootstrap и упрощает DevOps поддержку.

#### 3. **Хранение и повторное воспроизведение**
- Сообщения хранятся в логах, можно "перематывать" поток назад.
- Поддержка TTL, репликации и дискового хранения по темам и партициям.

#### 4. **Надёжность и семантика доставки**
- Поддерживает как *at-least once*, так и *exactly once* семантику.
- Благодаря `acks=all`, ретрай-логике и offset-менеджменту обеспечивается высокая отказоустойчивость.

#### 5. **Интеграции и совместимость**
- Kafka API поддерживается в большинстве брокеров.
- Отлично интегрируется с Hadoop, Spark, Flink, ClickHouse и множеством систем.
- Поддержка Connectors, REST Proxy, Schema Registry.

#### 6. **Развитая экосистема**
- Kafka Streams, KSQL, Kafka Connect дают возможности сложной стриминговой обработки без стороннего ПО.

#### 7. **Наблюдаемость**
- Поддержка Prometheus, JMX, Elastic, Grafana.
- Web-интерфейсы для DevOps: Confluent Control Center, Kafka Manager.

### Когда целесообразен выбор **RabbitMQ**

#### 1. **Традиционные очереди с маршрутизацией**
- RabbitMQ реализует модель "push-based" очередей с гибкой маршрутизацией сообщений через **обменники (exchanges)**.
- Поддержка типов обменников: `fanout`, `direct`, `topic`, `headers` — это мощный механизм для кастомной логики доставки.

#### 2. **Меньшая нагрузка и невысокие требования к throughput**
- Подходит для систем с умеренным количеством сообщений (до десятков тысяч сообщений в секунду).
- Не требует сложной настройки кластеров, партиций и логов.

#### 3. **Простота использования и администрирования**
- Быстрый старт, богатый UI (`RabbitMQ Management`), большое количество библиотек и tooling.
- Легко разворачивается в docker/k8s, есть готовые Helm-чарты и образы.

#### 4. **Поддержка множества протоколов**
- AMQP 0.9.1/1.0, MQTT, STOMP — позволяет интегрировать IoT, legacy-системы и мобильные приложения.

#### 5. **Транзакционность и гарантии доставки**
- Поддерживает ACK, NACK, retry, dead-letter queues.
- Позволяет более контролируемо обрабатывать сбои на уровне приложения.

#### 6. **Низкая задержка доставки**
- Практически мгновенная доставка сообщений (< 1–2 мс в пределах одного хоста).
- Применимо для real-time взаимодействия между микросервисами.

### Примеры кейсов, где RabbitMQ предпочтителен

- Обработка событий между микросервисами в небольших/средних проектах.
- Очереди задач для асинхронных worker-процессов (например, Celery в Python).
- Системы с критичной маршрутизацией (например, по ключам).
- Интеграция с системами, использующими AMQP или STOMP.
- IoT или mobile-сценарии, где нужен MQTT.

### Почему не другие:

- **NATS JetStream** — отличен для low-latency коммуникаций, но не подходит для аналитики и долговременного хранения.
- **Redis Streams** — удобно при наличии Redis, но сложно масштабируется и не обеспечивает строго гарантированной доставки.
- **ActiveMQ Artemis** — надёжный, но тяжёлый, более уместен в Java/Enterprise средах.

### Вывод

**Apache Kafka** — это промышленный стандарт для высоконагруженных, надёжных и масштабируемых систем обмена сообщениями. Благодаря отказу от ZooKeeper в новых версиях, она стала проще в эксплуатации, сохранив все преимущества масштабируемости и отказоустойчивости. Kafka — лучший выбор для архитектуры, в которой важны как скорость, так и возможность работы с "историей сообщений".
**RabbitMQ** — отличный выбор, когда приоритет — гибкость маршрутизации, простота и стабильность при умеренных нагрузках. Он не конкурирует напрямую с Kafka, а скорее дополняет его, закрывая те области, где Kafka может быть избыточен или сложен. Использование RabbitMQ может быть оправданным, особенно на ранних этапах проекта или в случаях, когда требуется быстрая интеграция и понятная эксплуатация.

> **Примечание 3**
>
> Из моего опыта Java Senior Backend-разработчика: чаще всего я работаю с Apache Kafka — брокер отлично подходит для масштабируемых систем с высокими нагрузками и сложной обработкой событий. RabbitMQ использую реже, но он удобен там, где важна простота настройки, гарантированная доставка и гибкие возможности маршрутизации.
>
> В реальных проектах Kafka и RabbitMQ — самые популярные брокеры сообщений, при этом Kafka практически стала стандартом де-факто построения event-driven архитектур.

### Есть три сервиса:

**minio**

**security**

**POST /v1/register**
1. Анонимный доступ.
2. Запрос направляется в сервис security POST /v1/user.

**POST /v1/token**
1. Анонимный доступ.
2. Запрос направляется в сервис security POST /v1/token.

**GET /v1/user**
1. Проверка токена. Токен ожидается в заголовке Authorization. Токен проверяется через вызов сервиса security GET /v1/token/validation/.

**POST /v1/upload**
1. Проверка токена. Токен ожидается в заголовке Authorization. Токен проверяется через вызов сервиса security GET /v1/token/validation/.

**GET /v1/user/{image}**
1. Проверка токена. Токен ожидается в заголовке Authorization. Токен проверяется через вызов сервиса security GET /v1/token/validation/.
