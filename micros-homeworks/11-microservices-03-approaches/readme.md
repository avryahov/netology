# Домашнее задание к занятию «Микросервисы: подходы»

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры.
Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.


## Задача 1: Обеспечить разработку

Предложите решение для обеспечения процесса разработки: хранение исходного кода, непрерывная интеграция и непрерывная поставка. 
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- облачная система;
- система контроля версий Git;
- репозиторий на каждый сервис;
- запуск сборки по событию из системы контроля версий;
- запуск сборки по кнопке с указанием параметров;
- возможность привязать настройки к каждой сборке;
- возможность создания шаблонов для различных конфигураций сборок;
- возможность безопасного хранения секретных данных (пароли, ключи доступа);
- несколько конфигураций для сборки из одного репозитория;
- кастомные шаги при сборке;
- собственные докер-образы для сборки проектов;
- возможность развернуть агентов сборки на собственных серверах;
- возможность параллельного запуска нескольких сборок;
- возможность параллельного запуска тестов.

Обоснуйте свой выбор.

### Решение

Для организации инфраструктуры CI/CD в облачной среде предлагается следующая архитектура:

**Системы и технологии:**
- **GitHub / GitLab (облачный)** — для хранения кода и управления репозиториями.
- **GitHub Actions / GitLab CI** — для настройки пайплайнов CI/CD.
- **Docker** — для создания и использования собственных образов.
- **HashiCorp Vault / GitHub Secrets / GitLab Variables** — для хранения секретов.
- **Kubernetes + Argo CD / Helm** — для автоматического деплоя.
- **Self-hosted Runners / GitLab Runners** — для собственных агентов сборки.

**Соответствие требованиям:**
- Облачная система: GitHub / GitLab Cloud.
- Система контроля версий Git: встроена в GitHub / GitLab.
- Репозиторий на каждый сервис: микросервисная структура.
- Запуск сборки по событию: Webhooks / push/pull request.
- Запуск по кнопке с параметрами: `workflow_dispatch` (GitHub) или Manual trigger (GitLab).
- Привязка настроек: `.github/workflows/` или `.gitlab-ci.yml`.
- Шаблоны сборок: шаблоны YAML или общие includes.
- Секретные данные: GitHub Secrets, GitLab Variables, Vault.
- Несколько конфигураций: разные jobs/stages в CI.
- Кастомные шаги: shell-скрипты и кастомные Docker-образы.
- Docker-образы: в CI конфиге или через Docker Hub / GHCR.
- Агенты на своих серверах: self-hosted runners.
- Параллельные сборки и тесты: matrix jobs и parallel pipelines.


**Обоснование выбора:**

На основе реального опыта в разработки со стороны Senior Java Backend разработчика, участие в преобразовании монолита к микросервисной архитектуре, написания bash-скриптов, корректиовки пайпланов могу с уверенностью предложить выше указанный стек технологий. Он является практически де-факто стандартом с массивным количеством микросервисов в корпоративной инфраструктуре разработки и поддержки конечного программного продукта. Мой выбор на Gitlab + Docker + Kubernates падает из-за большого количества документации, статей, реальных проектов. Их настройка, пользовательский интерфейс веб- и дескто-служб интуитивнее и понятнее нежели Jenkins + Docker SWARM. Хотя в защиту последних. Они мне попадаются не реже на малых и средних проектах. 

Предложенные технологии управления, сборки, доставки и развертки объективно и субъективно удобнее и популярнее на рынке разработки ПО. Соответствуем текущим трендам. 

## Задача 2: Логи

Предложите решение для обеспечения сбора и анализа логов сервисов в микросервисной архитектуре.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- сбор логов в центральное хранилище со всех хостов, обслуживающих систему;
- минимальные требования к приложениям, сбор логов из stdout;
- гарантированная доставка логов до центрального хранилища;
- обеспечение поиска и фильтрации по записям логов;
- обеспечение пользовательского интерфейса с возможностью предоставления доступа разработчикам для поиска по записям логов;
- возможность дать ссылку на сохранённый поиск по записям логов.

Обоснуйте свой выбор.

### Решение

Для сбора логов используется связка:

- **Vector** — как агент на хостах, собирающий логи из stdout.
- **Kafka** — как очередь доставки.
- **ClickHouse / Elasticsearch** — как хранилище логов.
- **Grafana / OpenSearch Dashboards** — для визуализации и поиска.

**Архитектура:**

1. Приложения пишут логи в stdout/stderr.
2. Vector собирает эти логи и отправляет в Kafka.
3. Из Kafka данные попадают в ClickHouse (через sink).
4. Grafana подключена к ClickHouse, либо используется OpenSearch Dashboards.

**Соответствие требованиям:**
- Сбор логов: Vector + Kafka.
- stdout: поддерживается Vector.
- Гарантированная доставка: Kafka с ack.
- Поиск и фильтрация: SQL / Lucene.
- UI для разработчиков: Grafana / OpenSearch Dashboards.
- Ссылки на поиск: доступно в Grafana или Dashboards.

**Обоснование выбора:**

Аналогично CI/CD выбору текущие инструменты распространены, я бы добавил LogBack, но решил указать Vector, так как он позволяет предварительно подготовить необходимый формат и отправить брокеру. Почему брокеру и почему кафке? В силу специфики работы и хранения данных очереди в топиках и партициях. Можно гарантировать доставку, сохранность данных. 

Почему OpenSearch / Grafana + Clickhouse? OpenSearch / Grafana как один из доступных сценариев. Можно напрямую использовать тройку богатырей "ELK", однако в свободном доступе на территории РФ легче OpenSearch задействовать. Можно тройку, но всё будет зависеть от сложности конечного ПО. 

Clickhouse часто начал видеть, как главный тренд по сборку аналитики. Не помешает бизнесу и менеджерам в обработке и прозрачности данных.

На первом этапе можно мой вариант, затем оставить кафку, подготовить тройку ELK, провести реверс, и с целью поддержки больших микросервисов от 100 и выше количеством будет пригодно. А первый стек, как вход и отслеживание прототипов, MVP-решений.

## Задача 3: Мониторинг

Предложите решение для обеспечения сбора и анализа состояния хостов и сервисов в микросервисной архитектуре.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- сбор метрик со всех хостов, обслуживающих систему;
- сбор метрик состояния ресурсов хостов: CPU, RAM, HDD, Network;
- сбор метрик потребляемых ресурсов для каждого сервиса: CPU, RAM, HDD, Network;
- сбор метрик, специфичных для каждого сервиса;
- пользовательский интерфейс с возможностью делать запросы и агрегировать информацию;
- пользовательский интерфейс с возможностью настраивать различные панели для отслеживания состояния системы.

Обоснуйте свой выбор.

### Решение

- **Prometheus** — сбор метрик.
- **Node Exporter / cAdvisor** — агенты для хостов и контейнеров.
- **Grafana** — визуализация и дашборды.
- **Alertmanager** — уведомления.
  
**Архитектура:**
- Node Exporter ставится на каждый хост (CPU, RAM, диск).
- cAdvisor собирает метрики с Docker-контейнеров.
- Приложения публикуют `/metrics` для бизнес-метрик.
- Grafana агрегирует и визуализирует.
- Пользовательские дашборды и алерты через Grafana UI.

**Соответствие требованиям:**
- Метрики всех хостов: через exporters.
- CPU, RAM, HDD, Network: метрики Node Exporter.
- Для каждого сервиса: cAdvisor + /metrics endpoint.
- UI и агрегирование: Grafana.
- Конфигурация дашбордов: JSON-дашборды, шаблоны.

**Обоснование выбора:**

Классика мониторинга и сбора данных. Дополнительно указал бы паттерн healthcheck внутри микросервисов, а затем их статусы через оркестратор или иной шлюз во вне прому с графаной передавал. 

В силу ограничения зоны ответственности на текущем атустафе не видел, но данный выбор применяется и используется постоянно на верхах как в крупных ритейл-системах, так и в низовых решениях. 

Популярность, документация сделали их распространенными
